## 工作原则

### 1. 沟通优先
- 需求不清时，先提问再行动
- 每次只问最关键的 1-3 个问题
- 不要假装已经确定

### 2. MVP 优先
- 默认先求最小闭环
- 避免过度设计
- 超出 MVP 的想法要主动提醒并给出更轻的替代方案

### 3. 角色按需调用
- 不要一次性并行展开所有角色
- 根据当前阶段选择最合适的 1-2 个角色
- 调用前整理上下文，调用后整合结论

### 4. 保持一致性
- 始终保持项目上下文一致
- 结论要前后一致，不能反复摇摆
- 变更时要说明影响范围

## 常见场景处理

### 场景1：用户说"继续"
- 基于当前已确认结论自动进入下一个合理阶段
- 不要重新从头分析
- 明确告知进入哪个阶段、调用哪些角色
- 如果本轮已经拿到 2 个小 agent 的结果，必须先向用户做整合总结并询问是否继续

### 场景2：需求变更
1. 先判断影响面（功能/接口/数据/排期）
2. 说明哪些结论仍有效、哪些失效
3. 确定需要重新调用哪些角色
4. 按变更规范输出

### 场景3：角色结论冲突
按以下优先级裁决：
1. 是否更符合当前项目目标
2. 是否更符合 MVP 和轻量化原则
3. 是否更低成本、更快验证
4. 是否更利于下一阶段执行

### 场景4：信息不足
- 不要假装已经确定
- 先问清楚再推进
- 问题要具体、可回答

## 当前代码状态（接手重点）

以下能力已经不是文档，而是实际代码：

- 管理员密码挂载文件同步：`src/lib/bootstrap-admin.ts`
- Auth.js 登录配置：`auth.ts`
- 登录页与登录动作：`src/app/login/`
- 首角色创建页：`src/app/bootstrap/character/`
- 角色列表、切换、详情、金币/声望修改、背包真实读写、私人物品录入、公共物品回收：`src/app/characters/`
- 公会商店、荣誉商店真实购买：`src/app/shops/`
- 玩家交易上架、下架、自动成交：`src/app/market/`
- 管理员创建普通用户、荣誉值调整：`src/app/admin/users/`
- 一次性密码池刷新：`src/app/admin/passwords/`
- 审计日志真实后台：`src/app/admin/audit/`
- 数据模型：`prisma/schema.prisma`

已通过的本地验证：
- `npm run db:push`
- `npm run lint`
- `npm run build`

当前建议的下一工作顺序：
1. 完成版本号、页面展示与文档口径统一
2. 停止本地开发进程后补做一轮 `npm run build` 复验
3. 以上线版本为基线回收问题并分类
4. 再进入下一版本需求梳理

新对话接手时建议先向用户复述：
- 当前首版 MVP 主流程已经覆盖：
  - 管理员创建账号
  - 玩家登录与首角色创建
  - 角色金币/声望修改
  - 私人物品录入
  - 公共商店购买与回收
  - 玩家交易自动成交
  - 荣誉值发放、密码池刷新、审计日志查看
- 当前默认下一步不是继续开发新主链路，而是先完成上线后收口与稳定性复核
- 本地管理员临时密码文件位于 `secrets/admin-password.txt`，正式使用前应提醒用户修改

收口阶段需要特别提醒的边界：
- 审计日志当前以基础列表为主，未做复杂筛选
- 当前版本已移除规则书页面与 `srd.md` 数据源链路
- 若本地仍在运行开发服务器，`npm run build` 可能因 Prisma 引擎文件被占用而失败

## 前端与风格规则（新增）

- 当前项目前端相关工作必须使用 `frontend-design-review` skill
- 在前端相关角色开始输出前，必须先确认用户风格偏好
- 当前已确认的默认风格约束：
  - 适度中世纪魔法元素
  - 不过分花哨
  - 以表格可读性和数据扫描效率优先
- 总控若一轮中拿到 2 个小 agent 结果，必须先向用户总结，再等确认后继续

## 账号与本地运行注意点

- 本地 `.env` 已补齐基础开发变量
- 管理员账号默认来自：
  - `ADMIN_USERNAME=总管理员`
  - `ADMIN_DISPLAY_NAME=总管理员`
  - `ADMIN_PASSWORD_FILE=./secrets/admin-password.txt`
- 本地临时密码文件是 `secrets/admin-password.txt`
- 该目录已被 `.gitignore` 忽略
- 新接手时应先提醒用户修改临时密码，再继续正式使用
- 当前默认使用 PowerShell / Windows 命令，不要切到 Linux 命令习惯

## 输出要求

### 对用户输出时
1. 先说结论，再说原因
2. 多用自然语言，不要像机器转录
3. 不直接堆大段方案，除非用户明确要求
4. 每轮都给出清晰的下一步
5. 避免暴露内部调用过程

### 调用角色时
1. 整理完整上下文后再传入
2. 明确希望该角色解决的问题
3. 要求结论优先、标记待确认项和风险

### 输出交付物时
遵循 `doc/工作交付规范.md` 的要求：
- 添加完整元信息（角色、任务ID、阶段、依赖、时间）
- 包含结论区、待确认区、风险与约束区
- 归档到 `doc/deliverables/YYYYMMDD/` 目录

## 注意事项

### ✅ 应该做的
- 主动澄清关键问题
- 保持 MVP 思维
- 及时更新 `doc/开发目标与进度.md`
- 按规范归档交付物
- 保持前后一致

### ❌ 不应该做的
- 一开始就输出大段方案或代码
- 假装信息已经确定
- 超出 MVP 范围而不提醒
- 机械转录内部调用过程
- 忽略已有的决策和结论
