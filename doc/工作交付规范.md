# Agent 工作交付规范

## 目的

确保项目总控与各下属 agent 之间的工作交付有迹可循，便于追溯决策过程、识别变更影响、保持上下文一致性。

---

## 1. 交付物标识规范

每次 agent 输出结果时，必须在文档头部添加以下元信息：

```
---
角色: [角色名称]
任务ID: [YYYYMMDD-HHmm-角色简称]
触发阶段: [需求澄清/方案设计/执行准备/变更评估]
依赖输入: [前置角色的任务ID，若无则填"无"]
输出时间: [YYYY-MM-DD HH:mm]
---
```

**示例：**
```
---
角色: 产品经理
任务ID: 20260310-1430-PM
触发阶段: 需求澄清
依赖输入: 无
输出时间: 2026-03-10 14:30
---
```

---

## 2. 标准输出结构

所有 agent 输出必须包含以下三个核心部分：

### 2.1 结论区（必需）
- 当前阶段的核心判断
- 明确的可执行建议
- MVP 范围界定

### 2.2 待确认区（必需）
- 影响推进的关键问题
- 需要用户或其他角色明确的事项
- 标记优先级（P0/P1/P2）

### 2.3 风险与约束区（必需）
- 当前方案的已知风险
- 技术/业务/时间约束
- 不做的事项及原因

---

## 3. 跨角色依赖标记

当某角色的输出依赖其他角色时，必须显式标记：

**格式：**
```
## 依赖关系
- 依赖 [角色名] 的 [任务ID]：[具体依赖内容]
- 阻塞 [角色名]：[本输出未完成前，该角色无法继续的原因]
```

**示例：**
```
## 依赖关系
- 依赖 产品经理 的 20260310-1430-PM：需确认核心用户场景
- 阻塞 前端开发：接口设计未完成前无法开始页面实现
```

---

## 4. 变更追溯机制

### 4.1 变更标记
当需求或方案发生变更时，新输出必须标记：

```
## 变更说明
- 变更类型: [需求变更/技术调整/范围收敛]
- 影响范围: [哪些角色的输出需要重新评估]
- 失效内容: [之前哪些结论不再有效]
- 保留内容: [哪些结论仍然有效]
```

### 4.2 版本关联
```
---
角色: 架构/API设计
任务ID: 20260310-1600-ARCH
触发阶段: 方案设计
依赖输入: 20260310-1430-PM
输出时间: 2026-03-10 16:00
替代版本: 20260310-1500-ARCH (因需求变更)
---
```

---

## 5. 总控整合规范

项目总控在整合多个角色输出时，必须输出：

### 5.1 整合摘要
```
## 本轮整合结论
- 调用角色: [角色1, 角色2, ...]
- 核心共识: [提炼的统一结论]
- 冲突处理: [如有冲突，说明裁决依据]
```

### 5.2 下一步动作
```
## 下一步
- 待用户确认: [需要用户决策的事项]
- 待调用角色: [下一阶段需要哪些角色]
- 可并行执行: [哪些任务可以同时进行]
```

---

## 6. 文档存档规则

### 6.1 存档位置
所有交付物按以下结构存档：

```
doc/
├── deliverables/           # 交付物归档
│   ├── YYYYMMDD/          # 按日期分组
│   │   ├── 任务ID-角色.md
│   │   └── ...
├── 工作交付规范.md         # 本文档
└── 开发目标与进度.md       # 进度追踪
```

### 6.2 归档时机
- 每次角色输出完成后立即归档
- 总控整合结论后归档整合文档
- 用户确认关键决策后归档确认记录

---

## 7. 质量检查清单

每次交付前，检查以下项：

- [ ] 是否包含完整的元信息标识
- [ ] 结论是否明确可执行
- [ ] 待确认项是否标记优先级
- [ ] 风险是否充分暴露
- [ ] 依赖关系是否显式标记
- [ ] 输出是否符合 MVP 原则
- [ ] 是否避免了超出当前阶段的内容

---

## 8. 反模式警示

**禁止以下行为：**

❌ 输出没有任务ID，无法追溯
❌ 结论模糊，无法直接执行
❌ 隐藏依赖关系，导致后续阻塞
❌ 变更时不说明影响范围
❌ 堆砌大量内容但没有明确下一步
❌ 超出 MVP 范围但不标记"可延后"
❌ 角色间结论冲突但总控未裁决

---

## 9. 示例：完整交付物

```markdown
---
角色: 架构/API设计
任务ID: 20260310-1600-ARCH
触发阶段: 方案设计
依赖输入: 20260310-1430-PM, 20260310-1500-TECH
输出时间: 2026-03-10 16:00
---

## 架构目标
- 当前要支撑的能力：用户登录与基础信息展示
- 推荐结构：前后端分离，RESTful API

## 模块划分
- 用户模块：负责登录、注册、信息查询
- 认证模块：负责 token 生成与验证

## 数据流
- 输入：用户名、密码
- 处理：验证 -> 生成 token -> 返回用户信息
- 输出：token + 用户基础信息

## API 设计
- POST /api/login：登录 / 返回 token 和用户信息
- GET /api/user/info：查询用户信息 / 需 token 验证

## 风险与约束
- 风险点：token 刷新机制暂未设计
- 边界待确认：是否需要第三方登录

## MVP 建议
- 现在先做：基础账号密码登录
- 后续再扩展：第三方登录、token 刷新

## 依赖关系
- 依赖 产品经理 的 20260310-1430-PM：已确认核心场景
- 依赖 技术经理 的 20260310-1500-TECH：已确认技术栈
- 阻塞 前端开发：接口设计完成后可开始实现

## 待确认（P0）
- token 有效期设置为多久？
- 是否需要记住登录状态？
```
