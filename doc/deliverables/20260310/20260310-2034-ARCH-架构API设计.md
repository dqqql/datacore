## 结论区

### 架构目标
- 当前要支撑的能力：
  - 管理员登录后台并管理账号、荣誉值、公共商店、一次性密码
  - 玩家登录后创建角色、维护角色金币和声望、查看背包、购买公共商品、进行玩家交易
- 推荐结构：
  - 单体全栈应用
  - Web 前端与 API 放在同一项目内
  - 关系型数据库负责交易一致性

### 推荐模块
- 认证模块：登录、会话、角色权限判断
- 用户模块：用户信息、荣誉值、账号状态
- 角色模块：角色创建、归档、恢复、当前角色切换
- 经济模块：金币、声望、荣誉值读写规则
- 背包模块：角色背包、私人物品创建、物品价格展示
- 公共商店模块：商店分类、商品管理、购买、半价回收、一次性密码校验
- 玩家交易模块：上架、下架、购买、物品转移
- 审计模块：金币/声望变更记录、管理员操作记录

### MVP 范围界定
- 先按明确业务规则建最小模块，不做可配置化规则引擎
- 先提供稳定接口，不做实时同步、消息通知和拍卖机制

## 模块划分

- 认证模块：处理用户名密码登录、角色权限、受保护路由
- 用户模块：保存账号名、显示名、荣誉值、账号角色
- 角色模块：保存角色基础信息、状态、归档标记
- 背包模块：保存角色持有的物品实例
- 商品目录模块：保存公共商店可售商品、分类、价格、货币类型、来源页
- 公共交易模块：处理商店购买与系统回收
- 玩家交易模块：处理私人物品上架、取消上架、成交
- 安全模块：处理公共商店条目修改所需的一次性密码池
- 审计模块：记录角色金币/声望调整和关键后台操作

## 数据流

### 数据流 1：登录与初始化
- 输入：用户名、密码
- 处理：认证成功 -> 返回当前用户信息 -> 查询是否已有角色
- 输出：登录态、账号荣誉值、角色列表、是否需要强制创建角色

### 数据流 2：公共商店购买
- 输入：角色 ID、商店商品 ID、数量
- 处理：校验角色归属 -> 校验余额 -> 扣减对应货币 -> 创建背包物品记录或增加数量
- 输出：购买结果、最新角色余额、最新背包数据

### 数据流 3：公共商店回收
- 输入：角色背包物品 ID、数量
- 处理：校验物品来源为公共物品 -> 计算半价 -> 扣减物品数量 -> 增加角色货币
- 输出：回收结果、最新角色余额、最新背包数据

### 数据流 4：玩家交易上架与成交
- 输入：卖方角色 ID、私人物品 ID
- 处理：校验物品可交易 -> 按当前价格创建上架单 -> 买方购买时扣款、转移物品、关闭上架单
- 输出：上架状态、成交结果、双方角色余额、双方背包变化

### 数据流 5：管理员修改商店条目
- 输入：商品变更内容、一次性密码
- 处理：校验管理员身份 -> 校验一次性密码有效 -> 更新商品 -> 写后台日志
- 输出：更新结果、密码使用结果

## API 设计

- 登录：`POST /api/auth/login`
  - 用途：账号密码登录
  - 核心字段：`username`, `password`

- 当前用户：`GET /api/me`
  - 用途：获取当前用户、荣誉值、角色摘要、是否需要创建首个角色

- 创建用户：`POST /api/admin/users`
  - 用途：管理员创建普通用户
  - 核心字段：`username`, `initialPassword`

- 创建角色：`POST /api/characters`
  - 用途：玩家创建角色
  - 核心字段：`name`

- 角色列表：`GET /api/characters`
  - 用途：查询当前用户全部角色及状态

- 更新角色数值：`PATCH /api/characters/:id/economy`
  - 用途：玩家更新本角色金币和声望
  - 核心字段：`gold`, `reputation`

- 归档角色：`POST /api/admin/characters/:id/archive`
  - 用途：管理员归档角色

- 恢复角色：`POST /api/admin/characters/:id/restore`
  - 用途：管理员恢复角色

- 背包列表：`GET /api/characters/:id/inventory`
  - 用途：查看角色背包

- 新建私人物品：`POST /api/characters/:id/inventory/private-items`
  - 用途：添加私人物品
  - 核心字段：`name`, `description`, `quantity`, `price`

- 商店列表：`GET /api/shops`
  - 用途：返回公会商店、荣誉商店及分类入口

- 商店商品：`GET /api/shops/:shopId/items`
  - 用途：查询某商店可售商品

- 商店购买：`POST /api/shops/purchase`
  - 用途：购买公共物品
  - 核心字段：`characterId`, `shopItemId`, `quantity`

- 商店回收：`POST /api/shops/sellback`
  - 用途：将公共物品半价卖回商店
  - 核心字段：`characterId`, `inventoryItemId`, `quantity`

- 玩家交易列表：`GET /api/market/listings`
  - 用途：查看所有在售私人物品

- 创建玩家交易：`POST /api/market/listings`
  - 用途：上架私人物品
  - 核心字段：`characterId`, `inventoryItemId`

- 取消玩家交易：`POST /api/market/listings/:id/cancel`
  - 用途：下架自己的物品

- 购买玩家交易：`POST /api/market/listings/:id/buy`
  - 用途：购买他人上架的私人物品
  - 核心字段：`buyerCharacterId`

- 管理员调整荣誉值：`POST /api/admin/users/:id/honor`
  - 用途：发放或扣减荣誉值
  - 核心字段：`delta`, `reason`

- 审计日志：`GET /api/admin/audit-logs`
  - 用途：查看金币、声望和关键后台操作记录

- 刷新一次性密码池：`POST /api/admin/shop-passwords/refresh`
  - 用途：重置公共商店修改密码池

- 批量导入公共商品：`POST /api/admin/shop-items/import`
  - 用途：预留批量导入入口
  - 核心字段：`shopId`, `items`

## 风险与约束区

### 风险点
- 市场成交和商店购买都涉及余额与库存双写，必须事务化
- 物品实例与商品定义必须区分，否则公共物品和私人物品会混用
- 一次性密码池如果与商店条目修改解耦不清，容易出现旧密码可重复使用

### 边界待确认
- 首版是否需要“当前选中角色”全局状态
  - 建议需要，减少页面反复选择角色
- 物品价格是否允许为 0
  - 建议允许，兼容奖励物品和占位物品
- 官方规则书物品页面是单独商店视图，还是商品来源筛选页
  - 首版建议先做筛选页而不是完全独立子系统

### 不做的事项及原因
- 不做通用规则引擎：当前规则稳定性不足
- 不做统一钱包：与“金币绑定角色”规则冲突
- 不做事件驱动架构：当前闭环完全可以用同步事务完成
