## 结论区
### 当前阶段核心判断
- 已确认 UI 改造方向采用方案 A：`Guild Archive`
- 本次改造以“在现有项目骨架上做可迁移的视觉与信息架构升级”为目标，不重做业务路由、不改变现有核心流程、不引入新的交互范式
- 当前最优策略不是立即改代码，而是先固定一份可执行的 UI 改造文档，待后续技术细节全部确认后，按文档分阶段实施

### 本次可执行建议
- 后续改造必须以当前信息架构为边界：左侧导航、顶部说明区、摘要卡片、表格列表、表单输入、状态提示为主结构
- 视觉方向保持“适度中世纪 / 奇幻账簿气质”，但必须明确以数据可读性和扫描效率优先，不能为了氛围牺牲表格、按钮、表单和反馈的清晰度
- 改造时优先重做共享外壳与通用组件，再重做业务页；不建议逐页各自发挥，否则风格会漂移

### MVP 范围界定
- 本文档只覆盖首版 UI 改造，不新增业务能力
- 本文档覆盖页面视觉、布局层级、组件样式、状态表现、响应式原则、迁移顺序、验收标准
- 不在本轮纳入：复杂动画系统、深度筛选器、批量操作工具栏、主题切换、多皮肤系统、品牌插画系统

## UI改造目标
### 改造目标
- 保留当前项目已形成的中世纪奇幻世界观语气
- 提升首页、控制台、角色页、市场页、商店页、后台页的一致性
- 提高表格、数字、状态与行动按钮的可读性
- 降低页面“装饰感压过信息”的风险
- 为后续统一实施提供可复用的样式规范和页面模板

### 不变的业务前提
- 登录后若无角色，仍需先创建角色
- 当前选中角色仍是全局核心状态
- 公共商店与私人交易仍保持分流
- 公共物品只允许系统商店流转
- 私人物品才允许进入玩家市场
- 管理后台仍保留独立入口和权限边界

### 本轮不改变的内容
- 路由结构
- Server Actions / 数据读写逻辑
- Prisma 数据模型
- 页面职责划分
- 权限规则
- 成功 / 失败的业务判断口径

## 设计总原则
### 总体原则
- 信息优先于装饰
- 操作优先于氛围
- 同类页面统一结构，不做风格漂移
- 列表页优先保证扫描效率
- 状态提示必须明确、稳定、可复用
- 同一个动作在不同页面保持同一视觉层级

### 方案A的视觉关键词
- 羊皮纸
- 铜饰边框
- 公会账簿
- 仪式感但克制
- 暖色纸面
- 深棕墨色文本
- 低饱和琥珀强调色

### 方案A的禁止事项
- 不使用高饱和金色大面积铺陈
- 不使用厚重纹理覆盖正文区
- 不在数据表格里加入过多装饰边框
- 不用过强阴影造成“卡片悬浮感过重”
- 不把标题字体用在大段正文或表格单元格里
- 不把状态信息只交给颜色表达

## 信息架构约束
### 必须保留的一级结构
- 左侧导航：玩家功能 / 公共商店 / 后台管理
- 主内容顶部头部区：页面标题、页面说明、阶段信息或关键提示
- 内容正文区：卡片、表格、表单、提示区组合

### 页面层级规则
- 一级层级：页面标题与页面用途说明
- 二级层级：摘要信息卡、当前状态、关键入口
- 三级层级：数据表格、表单区、规则说明区
- 四级层级：按钮、数值标签、辅助说明、状态标记

### 页面阅读顺序
- 先看页面标题和说明
- 再看当前角色 / 当前余额 / 当前规则
- 再看数据表格或主操作区域
- 最后处理次级说明和边界提示

## 视觉系统规范
### 色彩策略
- 主背景：暖羊皮纸色，接近 `#F4EAD7` 到 `#EFE3CB`
- 面板背景：比主背景更亮一阶，接近 `#FFF9F0`
- 主文本：深棕黑，接近 `#2B2117`
- 次级文本：偏灰棕，接近 `#6A5B4A`
- 强调色：琥珀棕，接近 `#7F5C2F`
- 强调色深阶：接近 `#5F421F`
- 成功色：低饱和深绿，接近 `#355F3B`
- 警示 / 失败色：低饱和陶土红，接近 `#A53F2B`
- 表格表头底色：强调色的极低透明度浅底，不可过深

### 用色比例
- 中性色和纸面色占 80% 以上
- 强调色只用于主要按钮、当前状态、重点标签、聚焦态
- 成功 / 失败色只用于状态反馈，不可挪作普通装饰色

### 背景策略
- 整页允许使用轻微径向暖光或纸张渐变
- 可保留极轻的纹理感，但只能在大背景层
- 内容容器内部禁止再叠加复杂图案

### 字体策略
- 标题字体：继续使用偏中世纪书卷感的衬线字体，适合标题和大数字摘要
- 正文字体：继续使用清晰的中文无衬线字体
- 标题字体只用于：
  - 页面主标题
  - 区块标题
  - 大数字摘要值
- 表格、表单、状态文本、按钮、说明文案全部使用正文无衬线字体

### 字号层级建议
- 页面主标题：36-48px
- 区块标题：24-30px
- 卡片数值：24-32px
- 正文说明：14-16px
- 标签和辅助说明：12-13px
- 表格正文：14px
- 表格表头：12px，允许全大写或较强字距，但必须保持易读

### 圆角与边框
- 页面主面板：28-32px
- 二级卡片：20-26px
- 表格外壳：20-24px
- 按钮：全圆角或 14-18px 圆角，当前项目建议主操作保持圆胶囊按钮
- 边框颜色必须偏柔和，避免纯黑硬边

### 阴影策略
- 使用浅而宽的暖色阴影
- 阴影只用于区分层级，不制造强悬浮感
- 重点依靠背景、边框和留白，而不是重阴影

## 通用组件改造规范
### AppShell
- 保留左侧导航 + 右侧主内容两栏结构
- 桌面端侧栏固定宽度建议 240-280px
- 右侧内容最大宽度建议控制在 1440-1500px 范围内
- 移动端改为单栏顺序布局，侧栏改顶部堆叠
- 侧栏头部必须有系统名称和一句简要说明
- 当前所在页面在导航中要有明显高亮，但不能抢过正文

### 页面头部 Header
- 包含页面 badge、标题、简短说明
- 可保留两个小型状态卡，展示“当前原则 / 当前阶段”这类静态信息
- 不堆过多按钮，头部区不承担复杂操作

### SummaryCard
- 结构固定为：标签 -> 数值 -> 说明
- 数值采用较大字号和较高对比
- 每行卡片说明不宜超过 2-3 行
- 卡片间距应统一，桌面端优先 3 列，平板 2 列，手机 1 列

### 表格容器 TableShell
- 外层必须包裹统一表格容器
- 表头采用浅强调底色
- 行高要足够，单元格上下内边距建议 14-16px
- 数字列右对齐，正文列左对齐
- 最后一列不能默认都右对齐，只有纯数值列才右对齐
- 行内动作按钮要控制数量，避免一行出现 3 个以上同权重按钮

### 按钮系统
- 主按钮：强调色实底，用于主要动作，如“进入系统”“立即购买”
- 次按钮：描边或浅底，用于“查看详情”“设为当前”
- 危险按钮：低饱和红描边或浅底，用于“归档”“下架”
- 禁用态必须降低对比，但仍保持可读
- 按钮文案必须是动词短语，不能只写“提交”

### 表单系统
- 标签始终显式显示，不使用仅 placeholder 的方案
- 输入框背景需比面板更亮，形成输入感
- 输入框边框要比普通卡片边框更清晰
- 聚焦态统一使用浅琥珀色 focus ring
- 错误提示与输入框之间保持稳定距离

### 状态提示组件
- 成功、失败、规则提醒三类提示统一结构
- 失败提示：浅红底 + 红色文案 + 柔和描边
- 成功提示：浅绿底 + 深绿文案 + 柔和描边
- 规则提示：浅棕或浅琥珀底 + 中性正文
- 提示组件不浮层化，优先出现在触发区域附近

## 页面级改造规范
### 1. 首页 `/`
#### 页面目标
- 作为项目总览与规则入口
- 告知系统当前阶段和固定规则
- 给出进入控制台的明确主入口

#### 结构要求
- 第一屏左侧：主标题、当前版本定位、项目一句话说明
- 第一屏右侧：上线前检查清单或当前关键提醒
- 第二屏：系统摘要卡片
- 第三屏：核心模块状态总表

#### 风格要求
- 首页是氛围感最强的页面之一，但仍不得牺牲信息密度
- 标题允许更强的中世纪气质
- 表格和清单区必须回归克制

### 2. 登录页 `/login`
#### 页面目标
- 让用户立即进入身份验证，不被装饰干扰
- 说明当前登录规则和首次进入后的角色要求

#### 结构要求
- 左侧：登录说明、规则提示、系统语气介绍
- 右侧：登录表单卡片
- 登录表单卡片宽度建议限制在 `420-520px`

#### 关键体验要求
- 用户进入后 3 秒内能明确知道要填哪里
- 错误提示要紧贴表单区域出现
- 主按钮必须是页内最强视觉焦点

### 3. 控制台 `/dashboard`
#### 页面目标
- 展示账号荣誉、当前角色、快捷入口
- 帮助玩家快速进入角色、商店、市场

#### 结构要求
- 顶部三张摘要卡：荣誉、当前角色、角色数量
- 左下：当前角色摘要
- 右下：快捷入口

#### 关键体验要求
- 当前角色必须足够醒目
- 快捷入口卡片保持统一高度和层级
- 不要在这里堆规则长文，控制台应该更偏“进入点”

### 4. 角色列表 `/characters`
#### 页面目标
- 管理账号下所有活跃角色
- 支持创建、切换、归档、查看详情

#### 结构要求
- 左侧主区：角色表格
- 右侧辅区：新增角色表单
- “当前角色”必须在表格中同时用文案与样式区分

#### 表格要求
- 列顺序建议保持：角色名称 -> 金币 -> 声望 -> 状态 -> 操作
- 当前角色状态不能只用颜色，需有文字标记
- 操作按钮主次分明：
  - 设为当前：次级高强调
  - 查看详情：普通次按钮
  - 归档：危险按钮

### 5. 角色详情 `/characters/[id]`
#### 页面目标
- 承载角色数值、背包、私人物品录入、交易挂单相关内容

#### 结构建议
- 顶部：角色基本信息 + 当前金币 + 当前声望
- 中部：背包表格
- 下部：私人物品录入 / 相关操作区

#### 关键要求
- 背包表格必须优先可读
- 私人物品与公共物品必须用结构和标签双重区分
- 若页面已有较多功能，优先做“分区标题 + 统一卡片”，不要做复杂折叠

### 6. 玩家市场 `/market`
#### 页面目标
- 让用户快速识别可购买挂单和自己的挂单
- 降低误买和误下架风险

#### 结构要求
- 顶部说明区必须明确当前角色和当前金币
- 中段规则提示要说明：
  - 只能整单购买
  - 不能买自己的挂单
  - 挂单不能在线改价
- 下方为主表格

#### 表格要求
- 我的挂单与可购买挂单必须文案上区分
- 购买按钮是主按钮，下架按钮是次或危险按钮
- 状态列不要只写“ACTIVE”，必须转成用户可读文案

### 7. 商店页 `/shops/guild` 与 `/shops/honor`
#### 页面目标
- 展示商店物品并完成购买
- 明确结算货币与规则

#### 结构要求
- 顶部：商店说明、当前角色、当前余额、规则摘要
- 下方：商店表格
- 行内包含数量输入和购买按钮，但布局必须整齐

#### 表格与操作要求
- 分类、物品、售价、数量、操作要清楚分栏
- 数量输入框宽度统一，避免在行内跳动
- 购买动作区不能挤压说明列
- 当前没有角色时，输入框和按钮禁用，同时在表格下有解释文本

### 8. 后台列表页
#### 范围
- `/admin/users`
- `/admin/shops`
- `/admin/passwords`
- `/admin/audit`

#### 页面目标
- 用统一后台视觉语言承载高密度信息
- 保持和玩家端同一主题，但更偏理性、更少装饰

#### 改造要求
- 后台页共用与前台一致的纸面、边框和字体系统
- 但后台页装饰强度要比首页和登录页更低
- 审计日志与后台表格优先考虑扫描效率
- 管理动作区与说明区要明显分开

## 状态设计规范
### 加载状态
- 页面级加载优先使用骨架或静态占位块，不建议整页大转圈
- 表格加载时保留表头，正文使用占位行
- 提交中的按钮文案应变化，如“正在验证身份...” “正在购买...”

### 空状态
- 无角色：引导创建角色
- 无挂单：说明当前暂无在售物品
- 商店无商品：说明尚未初始化或暂无条目
- 空状态文案必须附带下一步动作建议

### 错误状态
- 登录失败
- 购买失败
- 余额不足
- 商品已售出 / 已下架
- 角色不存在 / 不属于当前账户
- OTP 校验失败

### 成功反馈
- 购买成功
- 下架成功
- 归档成功
- 创建角色成功
- 密码池刷新成功

### 状态统一原则
- 所有成功 / 失败提示位置尽量稳定
- 同一页面的同类提示只保留一种视觉样式
- 任何关键错误都要说清“为什么失败”和“接下来做什么”

## 响应式规范
### 桌面端
- 保留双栏布局
- 导航侧栏常驻
- 表格完整展示

### 平板端
- 主内容优先单列或 2 列卡片
- 侧栏可上移为顶部块
- 表格如列过多，可保持横向滚动，但必须保留容器边界

### 手机端
- 全部改为单列
- 表格允许横向滚动，但操作按钮和状态必须仍可点击可见
- 顶部说明区和卡片区优先纵向排列

## 可访问性要求
- 文字对比度至少满足 WCAG AA
- 交互元素必须有可见 focus 样式
- 所有输入框必须有 label
- 颜色不能作为唯一状态表达
- 减少不必要动画，并遵守 `prefers-reduced-motion`

## 技术落地约束
### 必须遵守

- 尽量复用现有组件，不重复造大批新组件

### 可以接受的改动方式
- 扩展 `globals.css` 中的 CSS 变量和通用 class
- 调整 `AppShell` 与通用组件结构

### 不建议的方式
- 每个页面写一套独立风格
- 业务组件里大量硬编码颜色和阴影
- 为了视觉改造而调整业务路由或 Action 接口
- 在没有技术细节定稿前提前大改交互逻辑

- 首页与登录页保留中世纪账簿气质
- 玩家页与后台页气质统一但信息优先
- 没有出现一页一种按钮样式的情况

### 可读性验收
- 表格行距、列头、数字对齐清楚
- 卡片信息一眼能看懂
- 当前角色、当前余额、当前状态足够醒目
- 错误和成功反馈位置稳定

### 可迁移性验收
- 不改变现有主链路
- 不新增额外业务概念
- 样式层和组件层的改造可逐步落地
- 任意页面都能沿用同一视觉规范继续扩展

## 待确认区
- P0：最终是否继续沿用当前标题字体与正文字体组合，还是仅保留风格方向后换成其他同类字体
- P0：角色详情页是否需要在本轮一起做完整视觉改造，还是放到第二阶段
- P1：后台页是否保持和玩家页完全同壳，还是只共享设计 token、减少装饰层
- P1：是否需要在本轮补统一的空状态插图或仅使用纯文本空状态
- P2：是否要在正式改造时顺手补一轮页面微文案统一

## 风险与约束区
### 风险点
- 如果改造时跳过共享层，直接逐页改，会导致风格不统一且返工增加
- 如果给表格加入过多装饰，角色页、市场页、审计页的可读性会下降
- 如果首页和登录页的氛围感过重，会影响用户首次进入的操作清晰度
- 如果后台页沿用过强装饰，会削弱管理员高频操作效率

### 当前约束
- 项目已进入上线准备阶段，不适合在 UI 改造时同步改动业务规则
- 当前项目默认优先保证表格和数据可读性
- 当前项目仍需兼顾玩家端与后台端的一致性
- 后续所有实现都应以本文档为唯一 UI 基线，避免在改造过程中自行漂移

### 不做的事项及原因
- 不做新主题分叉：避免维护两套视觉系统
- 不做可配置主题切换：超出 MVP
- 不做营销站式重首页：当前项目是数据管理中心，不是宣传页
- 不做大量动画：不利于稳定和可读性
