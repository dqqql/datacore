# 服务器部署与版本更新指南

适用场景：
- 当前项目服务器运行在 Ubuntu 22.04
- 项目目录位于 `~/project/datacore`
- 应用通过 Docker Compose 管理
- 数据目录使用 `./prisma/sqlite`
- 密钥目录使用 `./secrets`

当前目标流程：
1. 本地开发并提交代码
2. 推送到 GitHub
3. 服务器进入项目目录执行 `git pull`
4. 执行 `docker compose up -d --build`

---

## 1. 当前部署约定

- Compose 文件：`docker-compose.yml`
- Docker 构建文件：`Dockerfile`
- 环境变量文件：`.env`
- 默认容器名：`datacore-ui`
- 默认对外端口：`3001`
- 容器内部端口：`3000`
- SQLite 挂载目录：`./prisma/sqlite:/app/prisma/sqlite`
- 密钥挂载目录：`./secrets:/app/secrets`

说明：
- 容器启动时会自动执行 `prisma db push`
- 这意味着功能改动如果引入 Prisma schema 变化，重建容器时会自动尝试同步数据库结构
- 但上线前仍建议先在本地完成 `npm run build` 验证

---

## 2. 服务器首次切换到 Compose 管理

如果服务器还在运行旧的手工 `docker run` 容器，例如 `datacore-ui-test`，按下面步骤切换：

```bash
cd ~/project/datacore

# 备份 SQLite 数据
cp -a prisma/sqlite "prisma/sqlite.bak-$(date +%F-%H%M%S)"

# 拉取最新代码
git pull --ff-only

# 先确认 compose 配置可解析
docker compose config

# 停掉旧容器
docker rm -f datacore-ui-test

# 启动新的 compose 服务
docker compose up -d --build
```

切换后验证：

```bash
docker compose ps
docker logs --tail=100 datacore-ui
curl -I http://127.0.0.1:3001
```

## 3. 后续每次版本更新流程

功能改动完成后，统一按下面流程上线：

### 本地

```bash
git status
npm run lint
npm run build
git add .
git commit -m "你的提交信息"
git push
```

### 服务器

```bash
cd ~/project/datacore
git pull --ff-only
docker compose up -d --build
docker compose ps
docker logs --tail=100 datacore-ui
```

如果服务正常，说明本次更新完成。

## 4. 功能改动时必须同步检查的配置项

如果后续功能修改涉及下面任一项，提交代码时必须一并更新相关文件：

### 4.1 环境变量变动

常见情况：
- 新增鉴权变量
- 新增第三方服务密钥
- 新增端口、域名、回调地址

需要同步检查：
- `.env`
- `.env.example`
- `docker-compose.yml`
- 相关代码中的环境变量读取逻辑
- 本文档中涉及的部署命令或说明

### 4.2 数据库结构变动

常见情况：
- 修改 `prisma/schema.prisma`
- 新增模型、字段、索引、枚举

需要同步检查：
- `prisma/schema.prisma`
- 本地 `npm run build`
- 容器启动时的 `prisma db push` 是否仍适用
- 是否需要额外数据迁移说明

### 4.3 端口或服务名变动

常见情况：
- 修改对外端口
- 修改容器名
- 新增反向代理或额外服务

需要同步检查：
- `docker-compose.yml`
- `.env.example`
- 服务器防火墙或反向代理配置
- 本文档中的验证命令

### 4.4 挂载目录变动

常见情况：
- SQLite 目录变化
- secrets 目录变化
- 新增上传文件目录

需要同步检查：
- `docker-compose.yml`
- `Dockerfile`
- `.dockerignore`
- 服务器目录结构

---

## 5. 上线前最小检查清单

每次准备上线前，至少确认：

- 本地 `npm run lint` 通过
- 本地 `npm run build` 通过
- 若改了配置，`.env.example` 已同步
- 若改了 Prisma schema，已确认启动时 `db push` 不会出意外
- 若改了挂载目录，`docker-compose.yml` 和服务器目录都已同步
- 已确认本次是否需要备份 `prisma/sqlite`

## 7. 当前已知边界

- 当前部署默认是单机 SQLite 方案，适合首版和轻量迭代
- 容器启动时自动 `db push` 适合当前 MVP 阶段，但后续如果数据库演进更复杂，可能需要切换到更严格的 migration 流程
- 当前文档默认端口为 `3001`，如果后续改为域名或反向代理，需要同步更新本文档

## 8. 反向代理与 8443 域名注意事项

如果外部正式访问地址不是默认的 `https://域名`，而是类似 `https://data.bg3ol.top:8443` 这种带端口的 HTTPS 域名，需要同时检查应用环境变量和反向代理转发头，否则 Next.js 16 的 Server Actions 会因为来源校验失败而拒绝登录请求。

### 8.1 应用侧必须同步的环境变量

示例：

```bash
AUTH_URL="https://data.bg3ol.top:8443"
NEXTAUTH_URL="https://data.bg3ol.top:8443"
SERVER_ACTIONS_ALLOWED_ORIGINS="data.bg3ol.top:8443"
```

说明：

- `AUTH_URL` / `NEXTAUTH_URL` 用于认证回调和应用基准地址
- `SERVER_ACTIONS_ALLOWED_ORIGINS` 用于允许反向代理后的安全来源
- 修改后需要重新执行 `docker compose up -d --build`

### 8.2 Nginx Proxy Manager 的关键检查项

如果使用 Nginx Proxy Manager 做反向代理，代理目标仍然指向应用容器暴露出来的 `3001` 端口即可，但要特别注意转发头中是否保留了外部端口。

本次问题的根因是：

- 浏览器 `Origin` 是 `data.bg3ol.top:8443`
- 代理转发给应用的 `X-Forwarded-Host` 变成了 `data.bg3ol.top`
- 两者不一致，Next.js 判定为非法 Server Actions 请求

建议至少确认：

- 代理条目的正式访问地址与实际登录地址完全一致
- `Host` 与 `X-Forwarded-Host` 转发后仍然包含 `:8443`
- 保存代理配置后执行一次 Nginx reload

### 8.3 若以后重建或升级 NPM 容器

如果未来重建、升级或迁移 Nginx Proxy Manager，需要重新验证一次 `data.bg3ol.top` 这条代理是否仍然保留了正确的 `Host` / `X-Forwarded-Host` 配置。

建议回归检查：

```bash
docker logs --since=2m datacore-ui
```

如果仍然出现类似下面的日志，优先检查反向代理，而不是先怀疑账号或数据库：

```text
`x-forwarded-host` header with value `data.bg3ol.top` does not match `origin` header with value `data.bg3ol.top:8443`
```
